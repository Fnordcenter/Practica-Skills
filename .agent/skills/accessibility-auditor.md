# Accessibility Auditor Skill

Use this skill whenever a webpage needs to be validated against web accessibility standards (WCAG guidelines) to ensure it is usable by individuals with disabilities.

## When to use this skill
- Whenever a user asks to "check accessibility," "audit a11y," or "make my site screen-reader friendly."
- After building a new interactive component (modals, dropdowns, tabs) that manages complex state changes.
- Prior to deploying a finalized webpage designed for a public or inclusive audience.

## Core Principles
- **Agnosticism:** Validate the generated DOM directly rather than relying solely on specific framework linters. Focus on standard HTML5 semantics and ARIA specifications.
- **Robustness:** A truly accessible site isn't just about passing automated tests; it's about logical document outlines (headers), focus management, and perceivable contrast.
- **Clarity:** Clearly state whether an issue is a Critical Blocker (e.g., empty buttons) vs. a Best Practice Refinement.

## How to use it (Step-by-Step)

### Phase 1: Analysis & Gathering
1. **Analyze Structure:** Scan the `index.html` structure to confirm standard page landmarks exist (`<header>`, `<nav>`, `<main>`, `<footer>`).
2. **Contextualize Interactivity:** Identify interactive components (buttons, links, form inputs) and dynamic UI changes (like popups or alerts).

### Phase 2: Scaffolding & Execution
1. **Semantic HTML Review:** 
    - Verify headings follow a logical structure (`<h1>` -> `<h2>` -> `<h3>`) without skipping levels.
    - Check that interactive elements use native tags (`<button>`, `<a>`) rather than `divs` with click handlers.
2. **Text Equivalents (Alt text):**
    - Ensure all `<img>` tags possess descriptive `alt` attributes or empty `alt=""` attributes if purely decorative.
3. **Contrast & Perception (CSS Analysis):**
    - Evaluate text color against background colors for adequate WCAG ratios (e.g., 4.5:1 for normal text).
    - Ensure focus states (`:focus`, `:focus-visible`) are prominent and visible for keyboard navigators.
4. **ARIA Implementation:**
    - Verify that any custom complex elements (e.g., custom dropdowns) use appropriate `role="listbox"`, `aria-expanded`, and `aria-hidden` attributes.
    - Check form labels (`<label for="input-id">`) correlate to input `id` attributes.

### Phase 3: Validation & Review
1. **Reporting:** Output an automated checklist or markdown report (`a11y-audit.md`) detailing:
    - Passed checks vs. failed checks.
    - Specific line numbers and DOM element references where violations occur.
    - Examples of how to structurally fix the violations (e.g., "Add `tabindex="0"` to allow keyboard focus").
2. **Delivery:** Provide the report to the user along with a code diff if explicit semantic structure refactors are accepted.

## Contingencies & Edge Cases
- **Non-Semantic DOM Constraints:** If the HTML is generated by a rigid third-party script that cannot easily be altered, recommend JS-based fixes (e.g., dynamically injecting ARIA attributes post-load).
- **High Contrast Requests:** If the user demands a specific branding color palette that fails WCAG standards, formally notify the user of the violation but respect the overriding request while suggesting a "High Contrast Mode" toggle.

## Specifications & Constraints
- Recommendations must prioritize Native HTML over ARIA Attributes whenever functionally possible ("No ARIA is better than bad ARIA").
- Must identify the absence of the HTML language declaration (`<html lang="en">`).
